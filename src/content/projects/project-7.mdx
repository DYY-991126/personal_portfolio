## 优化 Workflow 降低 90% 成本

在能力侧跑通后，我们发现生成一个设计稿要 **$1/page**。如果产品定价在 **$9 美元**，要保证不亏的情况下，只能给用户开放 9 个 Drafts 的生成额度。在真实工作中这点额度是不 work 的。对此，我通过优化 workflow 实现了成本优化，最终将成本降低至 **$0.1/page**。

### $1/page 的 workflow

页面先被拆成多个模块，**每个模块单独调一次生成**；且每次调用都带着**完整组件库**上下文。调用次数多、单次上下文又极大，token 很快堆到一页约 **$1**。

<ImageBlock
  src="/projects/project-7/成本优化/每个页面 $1.png"
  alt="阶段 1：单页约 $1 的 workflow"
/>

### $0.4-0.6/page 的 workflow

在仍按**模块多次调用**的前提下，改为只把**当前模块相关的组件**送进模型，单次调用变轻，一页落到约 **$0.4–$0.6**；但调用次数问题还在。

<ImageBlock
  src="/projects/project-7/成本优化/每个页面$0.4-$0.6.png"
  alt="阶段 2：单页约 $0.4-0.6 的 workflow"
/>

### $0.1/page 的 workflow

把生成粒度提到**页面级**：先做完页面级整理与组件筛选，再**每个页面只调一次生成**，且只注入该页真正用到的组件。调用次数与重复上下文一起降下来，单页约 **$0.1**。

## 实现细节

### 如何提高设计稿中的组件调用率

**定位问题**  
若结构阶段只有自然语言、执行阶段又塞入全量组件库，上下文既贵又杂，设计稿里很难稳定命中真实库件。

**解决方法**  
分三步：

**1. 向生成内容结构的 AI 提供裁剪后的组件库信息**  
来自 Figma 的原始组件库字段非常多，会占掉大量 token。我们只把组件的 **name** 与 **id** 提供给生成内容结构的 AI，用最少必要信息让它知道「真实组件库里到底有哪些组件」。

**2. 要求输出真实的 id 与 name**  
当生成内容结构的 AI 提出组件需求时，必须写出**真实的组件 id 与 name**，不能只用「要一个按钮 / 导航栏」这类描述。

**3. 生成设计稿的 AI 按需接收组件库信息**  
生成设计稿的第二个 AI 不会再接收整份组件库。而是会根据第一个 AI 组件清单按需注入。

### 如何让生成的设计稿效果更稳定展示？

**定位问题**  
生成的设计稿在画布内看起来有问题，去读 **DSL** 源码后又发现没问题。原因不是代码写错，而是**渲染时机**错了，导致元素之间重叠、挤压。

**解决方法**  
先等画布内的元素**全部渲染出来**，再把这些元素**应用上布局**。

### DSL 结构设计

面向 Canvas 落地的 **DSL** 以 `<dsl_spec>` 组织：包含布局容器（`div` / `childrenLayout`）、Design System **component**、Typography（`h1`–`p`）、以及用于素材注入的 **img**、**icon** 等节点类型与字段约定。
