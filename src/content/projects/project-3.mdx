## 背景

2025 年 9 月 Wegic 决定重做。在技术架构更换的大背景下，1.0 时期的各个功能在 2.0 都需要重新考虑。本项目相当于对 Wegic 2.0 进行了从 0 到 1 的产品设计。

## 项目挑战

1. 人力与时间资源打进，但要求在 2 周内快速交付
2. 团队内缺乏对于 Agent 产品的产品设计经验

## 应对策略

1. 协作策略上，我们尽量减少了人员节点。设计承担产品的工作，利用 cursor 制作代码设计稿交付开发。
2. 设计策略上，我们完全保留了 1.0 的视觉风格，主要聚焦于范围层到框架层。

---

## 功能与内容梳理与取舍

我对 Wegic 2.0 的产品设计命题理解：一个对小白用户易用的 AI 建站产品的交互体验是怎样的。许多功能生长于 1.0 的背景下，因旧技术架构的限制与产品高速迭代都使得产品走向复杂。要做到简单易用，对 1.0 的功能进行照搬是行不通的。我们的具体策略如下：

1. 梳理系统内全量功能，明确每个功能完成的用户任务及实现的用户目标。以对话交互为核心，衍生出一套交互体系。判断能被这套体系替代掉的 GUI 操作
2. 在新架构/产品设计下，对用户/业务没有价值的功能进行移除
3. 具备用户价值但需要重设计技术方案的功能，不纳入 2.0 mvp 范围。放入需求池后做优先级评估

## AI 对话流设计

### 理解 Agent 架构

#### Agent 架构的运作方式

用户输入进入主 Agent，在 Agent Loop 内按需调用 Tools，调用完成后回传主 Agent，无工具调用则输出最终回复。

<Project3AgentArchitectureMermaid />

#### 技术架构要点

1. **主 Agent（Kimmy）与 Tools**：用户消息进入 Kimmy；经 Function Calling、MCP 等挂载工具执行，并含 code-review 等分支；调用与收束由模型自主决定。
2. **assign-task 与子角色**：可并行委托 researcher、finder 等；与产品侧 Kimmy / Timmy 等角色呈现对应。
3. **Skills**：经 skill-enhance 接入 design、visual 等技能层（如 SKILL.md），对用户呈现为高阶任务语义。
4. **沙盒与运行特征**：生成与用户站点在沙盒隔离运行；ReAct（Thought → Action → Observation）多轮推理，工具与子任务可并行调度，流程非固定——执行顺序与前端展示不必一致，相对 1.0 流程化架构差异显著。

面对 Agent 架构，核心设计命题是：**AI 的运行过程不确定、并行、多工具调用——如何在前端让用户感知 AI 正在干活，又不被复杂的系统信息淹没？**

### 调研：信息暴露程度的光谱

我快速调研了 Cursor、Lovable、HeyBoss、Manus 等产品，按「系统运行信息暴露给用户的程度」来排列，可以画出一条从低封装到高封装的轴：

<GranularityAxis />

- **封装程度低**（如 Cursor）：把运行过程全量暴露给用户，每一步工具调用、推理过程都可见
- **封装程度高**（如 Lovable、HeyBoss、Manus）：对过程进行包装，只暴露少量、用户可理解的信息

### 从目标用户出发：我们该落在哪里

暴露运行过程固然能增加系统可见性，对建立用户信任有一定作用。但我的判断是：**我们的目标用户是小白用户，这个信任不需要在我们的产品内被构建。**

小白用户不需要看到、理解 AI 的内部运行。他们关心的是 AI 输出的结果，而非过程信息。我们与小白用户之间的信任，应该来自一次又一次输出的好结果，以及专业的服务体验——而不是把 AI 的内部运行细节摊开来给他们看。

因此，**Wegic 的封装程度落在封装程度高的一端。**

### 设计原则

1. **不主动展示全量过程**：不主动向用户展示 AI 处理任务的全量过程信息。
2. **只展示当前调用过程，并做概念包装**：只展示当前任务正在调用工具时的过程信息，并把这类信息用用户能理解的概念进行包装。让用户知道 AI 现在正在干活，而不是卡住了。
3. **提供可探索的入口**：当用户希望了解 AI 实际在干什么时，提供一个自然、明显的入口，让他们可以查看到 AI 正在做什么。

### 方案推导

#### 第一步：理解 AI 行为分类

明确原则后，第一件事是想清楚 AI 到底会产生哪几类行为。AI 的本质只有两类动作：

1. **普通文本回复**——无论是否在思考，输出都是纯文本，无需特殊设计。
2. **调用 Skill 或 Tool 做事**——这是设计重点。

<ChatPanelTwoTypesDemo />

对于第二类行为，需要解决「如何封装并呈现给用户」的问题。

#### 第二步：识别系统能力

先梳理清楚系统里有哪些 Tools 和 Skills：

| 类型 | 名称 |
| --- | --- |
| Tool | assignTaskTool、listDirectoryTool、readFileTool、writeFileTool、searchReplaceTool、genImageTool、editImageTool、genVideoTool、webSearchTool、deepResearchTool、managePlanTool、readLogTool、globTool、grepTool、execCommandTool、skill-enhance、firecrawl |
| Skill | design（website、effect）、visual（hero_video、launch_video、gen_logo） |

#### 第三步：封装——任务概念归类

按照设计原则，用户不需要知道 AI 调用了什么工具，只需要知道 AI 正在做什么**任务**。

我们将上述 Tools 和 Skills 按任务概念归类，当 AI 调用某个工具时，对外输出一个用户能理解的 description：

| 任务概念 | 对应工具 | 负责 IP |
| --- | --- | --- |
| 找文件 | readFileTool、grepTool、listDirectoryTool、globTool | Timmy、Turi |
| 做计划 / 更新计划 | managePlanTool | Kimmy |
| 生成图片 | genImageTool | Kimmy |
| 修改图片 | editImageTool | Kimmy |
| 生成视频 | genVideoTool | Kimmy |
| 做设计 | searchReplaceTool、execCommandTool、writeFileTool | Kimmy |
| 调研 | firecrawl、deepResearchTool | Kimmy、Turi |
| 测试 | readLogTool | Timmy |

#### 第四步：IP 与任务动画

每个任务概念对应一个专属 IP 形象，由它来负责这个任务的视觉呈现。IP 并非装饰性的，而是对底层 Agent 角色结构的映射——Kimmy 对应主 Agent（设计 / 产品方向），Timmy 对应工程类子角色，Turi 对应运营 / 研究类子角色。

每个 IP 在执行对应任务时，会播放与任务语义匹配的动画，让用户在不理解技术细节的情况下，也能直观感知到「正在有人在帮我做事」。

<TaskVideoGallery />

#### 第五步：外层卡片设计

至此，概念包装的底层逻辑已经完整。接下来要解决的是：**这些信息在卡片层面如何组织和呈现。**

用户发送一条指令后，AI 开始执行。从接到指令到任务结束，中途所有的工具调用都**封装在一个卡片内**。这个卡片的设计原则是：**让用户感知到 AI 正在运行，而不是卡住了；同时不用关心运行细节。**

卡片的具体呈现：

- **进度条 + 百分比**：表达整体任务进度
- **IP 头像 + 任务 description**：展示当前 AI 正在做什么，description 由 AI 调用工具时自动生成，用用户能理解的自然语言表述
- **结果物（Work Product）**：description 下方展示该任务的产出——做计划时展示待办面板，写代码时展示代码流，生成图片时直接展示图片
- **展开入口**：右侧提供展开按钮，进入 Wegic Studio 查看 AI 的全量运行记录

### Wegic Studio：AI 全量工作记录

虽然主要目标用户是小白用户，但数据观测中发现仍有一定比例的专业型用户。因此核心策略是：**默认直观，可选深入。**

点击卡片右侧的展开按钮，进入 Wegic Studio 面板。这里呈现的是 AI 从接到用户指令开始、到任务结束为止的**完整工具调用记录**——所有并发运行的任务、各自的执行状态，以及具体的结果产出。

## 项目结果

项目在规定时间内顺利上线。核心目标均已达成：界面框架完成简化，AI 对话流完整适配 Agent 架构。

上线后，Wegic 2.0 的核心业务转化表现显著优于 1.0——**核心转化指标较 1.0 版本提升超过一倍**。在资源有限、时间紧迫的条件下，聚焦核心体验、做出有效取舍，同样可以带来实质性的业务成果。
