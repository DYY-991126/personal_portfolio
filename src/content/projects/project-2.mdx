## Overview

这是 Wegic 2.0 里程碑冲刺阶段的核心执行项目之一。我作为项目负责人，主导了「生成前沟通」&「首次生成符合用户预期」系统的从零到一——在 AI 正式建站之前，先与用户逐步对齐诉求。**一期上线后，首次付款率从 4% 提升至 8%，实现翻倍。**

本项目承接 Project 1（目标定义与路径拆解），Project 1 定义了方向，这里记录的是子方向的具体执行过程。

---

## 背景：服务逻辑、里程碑与两个杠杆

### 第一层：产品逻辑——服务，而不是工具

市面上几乎所有的 Web Coding 类产品都是同一个模式：**用户输入需求 → 直接生成**。这是工具逻辑，假设用户知道自己要什么，且能清晰表达。

但真实的建站服务不是这样运作的。我在准备这个项目前，专门体验了国内外多个外包团队，发现他们有一个共同点：正式开工之前，都会先跟你聊一轮——了解你的业务、你的预期、你想用网站达成什么目标。然后基于这些，才出方案、开工。

这是服务逻辑。Wegic 的定位是「外包级建站服务体验」，应该遵循的是这个逻辑——主动了解客户、主动沟通、主动规划，而不是等用户下指令。

### 第二层：和里程碑的关系

里程碑目标是：**50% 的首次建站小白用户，在 1 小时内完成可对外发布的网站。**

要达成这个目标，有两个方向可以去发力：

1. **AI 首次生成的完成度**
2. **首次生成后的高效修改**

本需求聚焦在「首次生成」：通过与用户充分沟通建站目标、网站预期、所需内容和业务类型，收集详细信息；这些信息（包括内容结构和视觉要求）将用于指导 AI 生成，确保首次生成的网站与用户需求高度对齐，远超 Few-shot 生成的效果。

### 第三层：对其他需求的增益

这项工作不仅直接服务于里程碑，还为其他需求提供支撑。收集到的信息除了用于提升首次生成质量，还能支持「生成后的 AI 建议」和 Memory 等能力——生成网站后，AI 可为用户提供一条清晰的修改引导路径，以「网站发布」为最终目标，明确告知用户若要尽快发布，需要修改哪些步骤，实现最短的发布路径。

前提是：AI 在前置沟通阶段已经真正理解用户目标。**没有生成前沟通，生成后路径建议就会流于表面，用户也不会真正采纳。** 这是生成前沟通存在的根本价值——它既是提升首次生成质量的直接手段，也是让整条建站服务链路形成闭环的前提。

## 解题框架：预期的三个层次

要让首次生成就符合用户预期，需要先理解「预期」这件事本身。我们把它拆解成三个层次：

<Columns cols={3}>
  <div>
    #### 视觉层
    整体风格是否符合用户的审美预期——色调、排版、氛围感
  </div>
  <div>
    #### 内容层
    网站的信息架构是否服务于业务目标——页面数量、模块组成、说服逻辑
  </div>
  <div>
    #### 功能层
    表单、预约等后端能力是否到位
  </div>
</Columns>

想要这三层都符合用户的预期，那沟通的流程以及沟通的节点，就要围绕着这三层去展开。

## 产品设计方案：沟通流程大框架

### 设计思路：从三层到流程

<ProcessFlow variant="design" />

首先，我们要做的第一件事不是直接去和用户确认具体的内容结构（比如网站有什么功能、视觉效果如何），而是先和用户沟通需求。通过沟通，我们能明确用户对于网站的目标预期：他想用网站做什么？他的生意和业务逻辑是什么？在需求沟通阶段，通过两到三轮的对话，我们会形成一个大概的雏形。

基于这种沟通，我们明确了网站目标后，会针对目标设计一套内容结构。这个结构涵盖了网站的页面组成及每个页面的模块分布，主要包含两个层面：

1. **内容层面**：包括 Feature（功能特性）、客户评价、品牌背书、合作方的 Logo 墙等模块
2. **功能层面**：可能包含留资表单、日历预约等交互模块

确认内容结构后，我们会基于此生成六种风格的 9:16 视觉参考图，供用户选择。到这一步，内容层、视觉层和功能层就实现了匹配。

有了前置的需求了解、大的内容结构确认以及大方向的视觉偏好，下一步就是为网站填充真实信息。我们会给用户发送一个分门别类的信息收集表单，由用户填入已有的资料。

通过这三层的确认，我们能确保：(a) 大的内容结构是用户想要的；(b) 大的视觉风格符合用户偏好；(c) 网站细节中包含的是与用户业务强相关的真实信息。这样生成出来的网站，与用户预期的匹配度就会非常高。

### 流程总览

沟通流程围绕以下五个节点展开，每个节点对应一个「对齐预期」的必要动作：

```
需求沟通 → 内容结构确认 → 视觉风格确认 → 资料上传 → 生成网站卡片
```

| 节点 | 目标 |
| --- | --- |
| 需求沟通 | 了解业务背景与建站诉求，推导出网站所需的页面与模块 |
| 内容结构确认 | 基于诉求输出页面框架，供用户确认 |
| 视觉风格确认 | 用户确认结构后，生成 9:16 参考图供选择 |
| 资料上传 | 收集 AI 无法生成的核心事实信息 |
| 生成网站卡片 | 生成结果预览，引导至付费 |

## 通用性的沟通设计

我们为 AI 定了四条沟通原则：

1. **以用户业务为中心提问**
2. **灵活的沟通策略与流程**
3. **渐进式提问**
4. **控制消息长度**

以下逐一展开。

### 1. 以用户业务为中心提问

比较通用的询问方式通常是：「你想要什么类型的网站？」「你想要网站包含什么功能？」但我们不这么问。我们询问用户：**你的业务是什么？你的定位是什么？**

比如同样是香水业务，你是走性价比路线，还是走高端定制路线？再比如：「你想在网站上去卖你的产品，还是让别人联系你？」这个问题其实可以推导出用户的网站功能需求——是在线卖货还是留资获客——而不必直接问「你要不要支付」「要不要信息留资」这类技术选项。总之是围绕用户的生意去询问。用户是「业务专家」，对自己生意最熟悉，表达门槛最低。

映射到开场白上，我们不会问「你想做什么类型的网站」，而是用一种更松弛的方式邀请用户开口，比如：「想做个网站？先跟我说说你的想法呗，我们一起慢慢来。」不要求用户懂网站，只邀请他们谈自己的业务。从业务出发，才是最自然的起点。

### 2. 灵活的沟通策略与流程

初版方案把五个节点定义成严格的线性顺序。上线前测试时发现了问题：AI 对任何诉求都按部就班走完全部节点，缺乏判断力。一个好的沟通者不会这样——他会因人而异。

我们将五个节点重新定义为 AI 的「工具包」，不再依次执行，而是根据用户的信息量灵活判断：

- 用户给的信息越充分，AI 需要主动问的越少，可以更快进入生成
- 用户给的信息越稀疏，AI 就越需要用渐进式提问引导出来
- 无论用户怎么输入，AI 的目标始终是：在进入生成前，拿到足够的上下文

这个调整还解决了另一个实际问题：有些用户会多次对同一个环节表达不满意——比如视觉参考图不满意，AI 重新生成，再不满意，再生成。线性流程无法处理这类循环，工具包逻辑天然支持。

### 3. 渐进式提问

一个问题如果能合并，就要进行合并。能用一个问题问清的，就合并成一个，不要拆成多个小问题逐个问。反过来，也不一次性抛出所有问题让用户 overwhelm；该用渐进式引导时就用。核心是：根据当前上下文判断，能合并则合并。

### 4. 控制消息长度

不同 AI 的沟通体感差异很大。有的 AI 回复动辄几百字，用户反而抓不住重点。我们要求正文消息只包含 1–3 句话。为了降低用户的输入成本，我们增加了「智能建议」功能：当 AI 判断当前问答存在可选项时，底部会出现选项卡，用户点击即可，不必每次手动打字。同时支持多模态输入：图片、视频、文档直接上传，语音按住即说自动转文字填入输入框。整个沟通平均需要 6–7 轮才能进入生成流程。

## 每个节点的具体沟通策略

### 1. 需求沟通（Discovery）

**目标**：理解用户的业务，从而推导出网站需要哪些页面和模块。风格、素材、文案——都在后续阶段处理。

**策略要点**：

- 从用户的第一条消息出发——先提取他已经告诉你的信息，再决定问什么
- 问**生意**，不问**网站**——用户懂自己的业务，网站结构是 AI 的职责
- 从业务类型大胆推断——别问对品类而言显而易见的问题
- 每条问题都要过一关：**不同答案会不会改变页面结构？** 不会就跳过

**优先获取的结构性信息**：业务做什么、客户是谁、访客应该在网站上完成什么（购买 / 预约 / 联系 / 了解）、会改变架构的因素（电商、多门店、预约、多语言）等。

**反例**：物理店铺问「访客需要找到你的地址吗」；偏好类问题且不影响结构（联系方式放顶部还是底部）；听起来专业但结果相同的问题（「希望访客立即联系你还是先了解再联系？」——两种都需要内容展示 + 联系入口）。

**何时停止**：能设计出方向正确的内容结构即可。不必追求完美——具体结构更容易让用户反馈，阶段 2 的确认会做细化。

### 2. 内容结构确认（Content Structure Design）

**目标**：基于 Discovery 的结果，给出网站结构，交由用户确认后再往下走。

**三条原则**：

1. **需求驱动**：不套模板，完全基于用户想展示的内容和想达成的目标来设计。

2. **高效达成目标**：结构复杂度取决于说服成本——不同业务让访客「了解→行动」的成本不同。用户给的信息少时，若完全按用户说的做，结构可能对访客无效，所以要根据说服成本来补足。所有模块围绕目标，不推进目标的删掉，类似「如无必要，勿增实体」。

   | 举例 | 结构逻辑 |
   | --- | --- |
   | 简洁型（如摄影作品集） | 作品即证据，极简 |
   | 强信任型（如本地商家、SaaS） | 需要产品说明、Feature、评价等模块才能说服访客 |

3. **最小化**：在满足前两条的前提下，用最少的页面和模块。业务上更快更省成本；用户上修改负担更小——同一目标，6–7 页和 2 页都能完成时，2 页更好。

---

**实操中的思考**：视觉风格图本身带有内容结构，是否还需要单独这一步？需要。视觉图通常只展示单页，多页网站单图覆盖不了；视觉模型不擅长按业务逻辑规划结构；且我们验证过，让模型基于这些原则生成结构，比跳过该节点，后续结构更符合业务、更高效。

### 3. 视觉风格确认（Style Selection）

我们选择在付费前给用户展示 9:16 的视觉参考图，背后有几层考量：**建立信任**（用户第一次看到具体交付物预期，整个流程的第一个 Aha Moment）、**达成视觉共识**（「简约」「温馨」因人而异，用图说话更准）、**技术与成本的最优解**（生图比直接生成完整网站更优，尤其当用户尚未确定风格时）。

风格参考图会根据内容结构的首页进行生成。例如，内容结构首页有 6 个模块，生成的视觉参考长图就对应有 6 个模块。实现流程：先调用工具，基于内容结构 JSON 和需求沟通信息生成 6 份生图提示词；再将这 6 份提示词分别交给 6 个 NanoBanana2 实例，并发执行，以加速生成。

**风格生成的核心要求**：AI 不能为了凑齐 6 种风格而选择不适合用户业务的风格，而应根据用户业务挑选合适风格。若用户业务只适合 2–3 种风格，6 张图应在这 2–3 种风格内做变体——例如同样是简约风，可以有 2 种不同感觉的简约、3 种不同感觉的清新；不会为凑齐 6 种截然不同的风格就突然出现赛博朋克风。

现实边界：AI 无法 100% 还原参考图。策略是接受 7–8 成还原度，在 UI 上说明这是「风格参考图」，而非最终还原稿。

### 4. 资料上传（Asset Collection）

有了内容结构和视觉风格，网站的大框架就定下来了。接下来要填充具体信息——联系方式、产品图、Logo 等。这里有一个关键区分：有些信息客观存在、只有用户有，AI 无法捏造；有些信息是表达层面的，AI 可以代劳。我们的原则是：**收集事实，而非表达**。

- **事实**：客观存在、无法捏造（联系方式、实拍照片、价格、Logo、营业时间）——必须向用户收集
- **表达**：AI 能生成或润色（Slogan、品牌理念、介绍文案、模块标题）——不向用户索取

**收多少——只收不可替代的**：

- 优先 AI 无法生成的内容：用户自己的产品图、Logo、团队照、真实评价
- 不收 AI 能生成的：装饰图、背景图、通用插画
- 若前期对话已有足够素材，不再重复收集

**语气**：把收集说成「让网站更像你自己」，而不是作业。强调可以跳过，AI 可生成替代内容。

### 5. 生成网站卡片

生成结果预览，引导至付费。用户确认后进入建站。

### 边界处理

对于我们无法实现的网站类型（如重后端的后台管理系统），Prompt 中明确：若技术上不可行，**如实说明，并给出替代方案**，由用户决定。避免用户付费、生成后才发现做不到。除品牌考量外，还有现实成本：每笔客诉银行判罚 $15——来自市场侧同学的反馈。

## 落地过程

### 发起方式

这个项目脱胎于 Project 1 的优先级评估——P0 方向 (b) 即「首次生成符合用户预期」。

方向确定后，我在内部群里公开发起，感兴趣的同学自愿加入。启动顺序是：先拉设计同学，因为我们可以直接用 Cursor 做出一个 Demo，有具体的东西更容易对齐；Demo 做出来后，再拉前端和能力侧开发进来。说服过程比较顺畅——说清楚想做什么、为什么要做，询问档期，大家积极性都比较高。

**核心成员：** 我（项目负责人）、设计师 A（上传模块等）、设计师 B（专职 Prompt 调优）、前端开发、能力侧开发

**时间线：** 2025.12.15 前后启动，2026.1.9 一期上线，历时约一个月。

### 上线后遇到的两个卡点

**卡点一：积分成本怎么定**

对话功能需要消耗用户积分，上线前需要确定初始额度。我们测算了整个生成前沟通流程的积分消耗，约为 60–70 分；在此基础上乘以 2 预留冗余，最终给用户分配了 **150 积分**的初始额度。

**卡点二：Prompt 交接摩擦**

将写好的 Prompt 交接给 AI 能力侧时，我们多次发现内容被改动。能力侧认为是无关紧要的细节，但从业务角度这些改动有实质影响——他们的视角是技术实现，而不是用户体验。结果是 Prompt 调整来回耗费了大量时间。

解决方法是在 Prompt 里明确分层标注：哪些是业务强要求、不可改动；哪些是技术实现层面、能力侧可自行决定。分层清晰之后，摩擦明显减少。

### 从一期到二期

一期上线后，我们持续观察线上数据和用户行为，推动了二期迭代。二期的改动，一部分来自原始规划，一部分是线上观察到的新问题。

**架构升级：双 AI → 模式切换**

一期方案是新起一个独立的「沟通 AI」，沟通结束后将上下文总结成卡片，传递给「生成 AI」。

问题在于：上下文传递必然有信息损失。在 context 没有爆掉的情况下，同一个 AI 处理始终优于跨模型交接。

二期改为单 AI 的**模式切换**：在同一个 AI 内设定需求沟通模式和生成代码模式，上下文完整保留，生成质量更稳定。

**从 Structured Output 到 Tool Call**

一期中，AI 在输出内容结构草图和视觉参考图时，使用的是 Structured Output。上线后发现输出不稳定——内容有时不显示，或出现解析错误。

我在一篇解析 Claude 架构的国外文章中看到了 Tool Call（工具调用）方案，在会议上提出，工程侧和 PM 侧都认可——其实大家之前就知道这个方案更好，只是出于成本原因没有先做。切换后，我们专门为此做了一套 AI UI 工具，稳定性和速度均有提升。这也成为后续的方向：所有 AI 输出的非正文类消息，统一改用 AI to UI 方式封装，全部走 Tool Call。

**内容结构确认（二期新增）**

一期没有内容结构确认环节。二期加入后，发现它对生成质量的影响远比预期大。

此前使用 Gemini3 时，AI 容易「偷懒」，只生成少量页面内容。当时我们在 Prompt 里做了硬性规定——必须生成多页，每页 7–8 个模块。现在去掉这些硬性限制，改由内容结构确认来保证质量。内部评估后，这种方式的生成结果是过关的。

**上传流程重设计**

一期中 AI 用纯文本提示用户上传素材，体验较散。二期改为 Markdown 表单——左侧展示素材类型，右侧为控件，结构更清晰。每项新增「AI 帮我生成」按钮，用户没有该项资料时可由 AI 代为填充。流程出口也增加了「暂时跳过」和「提交素材」两个选项，用户不必为缺料卡住。

---

## 结果

<Callout type="insight" title="一期数据">
  一期上线后，首次付款率从 **4% 提升至 8%**，近乎翻倍。
</Callout>

我判断提升的关键是**信任感的建立**。视觉参考图是整个流程里用户的第一个 Aha Moment——在这之前，用户只是在和 AI 聊天；参考图出现的那一刻，用户第一次看到了一个具体的交付物预期，信任门槛随之降低。

有一个有趣的数据插曲：直接发长 Prompt 的用户，付费率略高于「聊出来」的用户。推测是因为这类用户在项目上投入的精力更多，付费意愿天然更强。工具包逻辑的好处在于它同样适配这类用户——AI 拿到充分信息后，直接跳过沟通进入生成。

二期的核心目标是提升生成质量，对应指标是用户的编辑次数和编辑时长的前后变化。由于我在数据充分积累前离开了公司，没有拿到具体数值。
