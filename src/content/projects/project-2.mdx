## Overview

这是 Wegic 2.0 里程碑冲刺阶段的核心执行项目之一。我作为项目负责人，主导了「生成前沟通」系统的从零到一——在 AI 正式建站之前，先与用户逐步对齐诉求。**一期上线后，首次付款率从 4% 提升至 8%，实现翻倍。**

本项目承接 Project 1（目标定义与路径拆解），Project 1 定义了方向，这里记录的是具体的执行过程。

---

## 背景：里程碑与两个杠杆

里程碑目标是：**50% 的首次建站小白用户，在 1 小时内完成可对外发布的网站。**

要达成这个目标，我们拆解出两个主要发力方向：

<Columns cols={2}>
  <div>
    #### 首次生成质量
    确保首次生成的网站在整体方向上符合用户的业务目标，减少后续修改量。这是本项目的核心。
  </div>
  <div>
    #### 高效的修改方式
    支持用户在生成后高效修改——圈选元素直接评论、版本回退兜底负向体验。与本项目并行推进。
  </div>
</Columns>

连接两个方向的是**生成后 AI 引导**：AI 在生成完成后，基于用户目标主动规划一条最短发布路径——帮用户梳理接下来改什么、按什么顺序改，可以最快达到可发布状态。

这条路径要真正有效，前提是 AI 已经了解用户的目标。**如果没有生成前沟通，AI 对用户的业务背景一无所知，生成后的路径建议就会流于表面，用户也不会真正采纳。** 这是生成前沟通最根本的存在价值——它不只是为了提升首次生成质量，更是为了让整条建站服务链路形成闭环。

---

## 核心洞察：服务逻辑 vs 工具逻辑

市面上几乎所有的 Web Coding 类产品，都是同一个模式：**用户输入需求 → 直接生成**。这是工具逻辑，它假设用户知道自己要什么，并且能够清晰地表达出来。

但真实的建站服务不是这样运作的。我在准备这个项目前，专门体验了国内外多个外包团队，发现他们有一个共同点：正式开工之前，都会先跟你聊一轮——了解你的业务、你的预期、你想用网站达成什么目标。然后基于这些，才出方案、开工。

这是服务逻辑。Wegic 的定位是「外包级建站服务体验」，应该遵循的是这个逻辑——主动了解客户、主动沟通、主动规划，而不是等用户下指令。

---

## 解题框架：预期的三个层次

要让首次生成就符合用户预期，需要先理解「预期」这件事本身。我们把它拆解成三层：

<Columns cols={3}>
  <div>
    #### 内容结构层
    网站的信息架构是否服务于业务目标——页面数量、模块组成、说服逻辑
  </div>
  <div>
    #### 视觉层
    整体风格是否符合用户的审美预期——色调、排版、氛围感
  </div>
  <div>
    #### 功能层
    表单、预约等后端能力是否到位
  </div>
</Columns>

这个项目重点解决**内容结构层**和**视觉层**。功能层由并行项目 Wegic Cloud 负责推进——两个方向共同支撑最终的里程碑目标。

## 沟通流程设计

### 入口：开一个好问题

整个沟通从一个开放性问题开始：

> "你想构建什么？可以跟我说，然后我会一步一步带着你，帮你完成网站。"

这个设计有意义：它不要求用户懂网站，只邀请他们谈自己的业务。我们的用户是「业务专家」而非「网站专家」——对自己的生意最熟悉，表达门槛最低。从业务出发，才是最自然的起点。

### 五个核心节点

沟通流程围绕以下五个节点展开，每个节点对应一个「对齐预期」的必要动作：

```
需求沟通 → 内容结构确认 → 视觉风格确认 → 资料上传 → 生成网站卡片
```

| 节点 | 目标 |
| --- | --- |
| 需求沟通 | 了解业务背景与建站诉求 |
| 内容结构确认 | 基于诉求输出页面框架，供用户确认 |
| 视觉风格确认 | 生成 9:16 参考图，供用户选择风格 |
| 资料上传 | 收集 AI 无法生成的核心事实信息 |
| 生成网站卡片 | 生成结果预览，引导至付费 |

### 关键设计调整：从线性流程到工具包逻辑

初版方案把五个节点定义成严格的线性顺序。上线前测试时发现了问题：AI 对任何诉求都按部就班走完全部节点，缺乏判断力。一个好的沟通者不会这样——他会因人而异。

我们将五个节点重新定义为 AI 的「工具包」，不再依次执行，而是根据用户的信息量灵活判断：

- 用户给的信息越充分，AI 需要主动问的越少，可以更快进入生成
- 用户给的信息越稀疏，AI 就越需要用渐进式提问引导出来
- 无论用户怎么输入，AI 的目标始终是：在进入生成前，拿到足够的上下文

这个调整还解决了另一个实际问题：观察到有些用户会多次对同一个环节表达不满意——比如视觉参考图不满意，AI 重新生成，再不满意，再生成。线性流程无法处理这类循环，工具包逻辑天然支持。

## 沟通策略

流程定好了，AI 具体怎么说话，同样重要。

不同 AI 的沟通体感差异很大。有的 AI 回复动辄几百字，吐出大量信息，用户反而抓不住重点。我们为 AI 定了四条沟通原则：

1. **渐进式提问**：不一次性抛出所有问题
2. **一次只问一个问题**：降低用户认知负担
3. **控制消息长度**：正文消息只包含 1–3 句话
4. **以用户业务为中心**：问生意和目标，不问技术选项

整个沟通平均需要 6–7 轮才能进入生成流程。为了降低用户的输入成本，我们增加了「智能建议」功能：当 AI 判断当前问答存在可选项时，底部会出现选项卡，用户点击即可，不必每次手动打字。

同时支持多模态输入：图片、视频、文档直接上传，语音按住即说自动转文字填入输入框。一个典型场景是外包客户把整理好的需求文档直接发进来，AI 读取后直接提炼，跳过逐轮问答。

## 各节点设计细节

### 内容结构确认

视觉风格图（9:16）本身含有内容结构，有人会问：这个节点是必要的吗？

是的，原因有两点。第一，视觉风格图通常只展示单页，但实际网站可能是多页的，单页图无法覆盖完整业务需求。第二，我们的视觉生成模型（Nano Banana 2）能生成美观的参考图，但它不具备依据业务逻辑规划内容结构的能力——这两件事在技术上是分开的。

内容结构有一条核心设计原则：**在高效达成用户业务目标的前提下，采用最精简的网站逻辑。**

背后是一个关于「说服成本」的判断：

| 网站类型 | 结构逻辑 |
| --- | --- |
| 简洁型（如摄影作品集） | 核心是展示作品，结构极简 |
| 强信任型（如本地商家、SaaS） | 需要产品说明、Feature、客户评价等模块才能说服访客 |

不同诉求对应不同的必要模块量。生成多了是浪费，也增加用户后期修改的负担；生成少了达不到目标。刚好够用的结构，才是真正对用户有利的设计。

### 视觉风格确认

我们选择在付费前给用户展示一批 9:16 的视觉参考图，背后有几层考量：

**建立信任**是首要动机。在没有看到任何实物的情况下，用户很难产生付费的信心。参考图出现的那一刻，用户第一次看到了具体的交付物预期——这是整个流程里用户的第一个 Aha Moment。

**达成视觉共识**是第二层。「简约」「温馨」这类形容词，每个人的理解天差地别。用图说话比用文字说话准确得多。

**技术与成本的最优解**是第三层。先用生图模型展示网站的样子，比直接生成完整网站在技术路径和成本上都更优——尤其是在用户不确定自己要什么风格的情况下。

这里有一个现实的边界需要管理：AI 不能 100% 还原参考图。追求极致还原需要工程侧单独解析，成本高且方案复杂。我们的策略是接受 7–8 成的还原度，并在 UI 文案上统一说清楚：这是「风格参考图」，而非最终还原稿。

我们也尝试过把 9:16 的长图切割成多张单图传给 AI 以提升还原性；但综合 token 消耗和未来模型迭代成本，最终保留了传整图的方案——随着 Gemini 3.1 在视觉理解方面的持续迭代，还原度会自然提升。

### 资料上传（AI to UI 表单）

很多产品的上传设计，是「需要什么素材就叫用户去上传什么」。我们的出发点不同：**收集的是 AI 无法生成的信息，而不是所有信息。**

这里有一个清晰的分类：

| 类型 | 举例 | 由谁处理 |
| --- | --- | --- |
| 事实层面（用户必须提供） | 电话、邮箱、地址 | 用户 |
| 宣传层面（AI 可以代劳） | Slogan、文案介绍、Logo | AI |

因此我们有三条收集原则：收集最少量的资料；这些资料必须是 AI 无法生成的；且在网站内是必要的。

表单按信息类型分类展示，每一项都有「AI 帮我生成」的选项——用户没有该项资料时，点击标记，AI 代为填充。

### 边界处理

对于我们无法做到的网站类型（如重后端的后台管理系统），我们在 Prompt 里做了明确声明：如果用户的诉求在技术上无法实现，**诚实告知，并给出平替方案**，由用户自行判断。

这避免了用户充完钱、生成完，才发现我们做不到。除了品牌口碑的长期考量，还有一个很现实的数字：每一笔客诉，银行判罚 $15——这是市场侧同学在处理客诉时同步过来的。

## 落地过程

### 发起方式

这个项目脱胎于 Project 1 的优先级评估——P0 方向 (b) 即「首次生成符合用户预期」。

方向确定后，我在内部群里公开发起，感兴趣的同学自愿加入。启动顺序是：先拉设计同学，因为我们可以直接用 Cursor 做出一个 Demo，有具体的东西更容易对齐；Demo 做出来后，再拉前端和能力侧开发进来。说服过程比较顺畅——说清楚想做什么、为什么要做，询问档期，大家积极性都比较高。

**核心成员：** 我（项目负责人）、设计师 A（上传模块等）、设计师 B（专职 Prompt 调优）、前端开发、能力侧开发

**时间线：** 2025.12.15 前后启动，2026.1.9 一期上线，历时约一个月。

### 上线后遇到的两个卡点

**卡点一：积分成本怎么定**

对话功能需要消耗用户积分，上线前需要确定初始额度。我们测算了整个生成前沟通流程的积分消耗，约为 60–70 分；在此基础上乘以 2 预留冗余，最终给用户分配了 **150 积分**的初始额度。

**卡点二：Prompt 交接摩擦**

将写好的 Prompt 交接给 AI 能力侧时，我们多次发现内容被改动。能力侧认为是无关紧要的细节，但从业务角度这些改动有实质影响——他们的视角是技术实现，而不是用户体验。结果是 Prompt 调整来回耗费了大量时间。

解决方法是在 Prompt 里明确分层标注：哪些是业务强要求、不可改动；哪些是技术实现层面、能力侧可自行决定。分层清晰之后，摩擦明显减少。

### 从一期到二期

一期上线后，我们持续观察线上数据和用户行为，推动了二期迭代。二期的改动，一部分来自原始规划，一部分是线上观察到的新问题。

**架构升级：双 AI → 模式切换**

一期方案是新起一个独立的「沟通 AI」，沟通结束后将上下文总结成卡片，传递给「生成 AI」。

问题在于：上下文传递必然有信息损失。在 context 没有爆掉的情况下，同一个 AI 处理始终优于跨模型交接。

二期改为单 AI 的**模式切换**：在同一个 AI 内设定需求沟通模式和生成代码模式，上下文完整保留，生成质量更稳定。

**从 Structured Output 到 Tool Call**

一期中，AI 在输出内容结构草图和视觉参考图时，使用的是 Structured Output。上线后发现输出不稳定——内容有时不显示，或出现解析错误。

我在一篇解析 Claude 架构的国外文章中看到了 Tool Call（工具调用）方案，在会议上提出，工程侧和 PM 侧都认可——其实大家之前就知道这个方案更好，只是出于成本原因没有先做。切换后，我们专门为此做了一套 AI UI 工具，稳定性和速度均有提升。这也成为后续的方向：所有 AI 输出的非正文类消息，统一改用 AI to UI 方式封装，全部走 Tool Call。

**内容结构确认（二期新增）**

一期没有内容结构确认环节。二期加入后，发现它对生成质量的影响远比预期大。

此前使用 GPT-3.5 时，AI 容易「偷懒」，只生成少量页面内容。当时我们在 Prompt 里做了硬性规定——必须生成多页，每页 7–8 个模块。现在去掉这些硬性限制，改由内容结构确认来保证质量。内部评估后，这种方式的生成结果是过关的。

**上传流程重设计**

| 项目 | 一期 | 二期 |
| --- | --- | --- |
| 形式 | AI 发纯文本提示 | Markdown 表单（左侧素材类型，右侧控件） |
| AI 代填 | 无 | 每项有「AI 帮我生成」按钮 |
| 流程出口 | 无跳过选项 | 新增「暂时跳过」和「提交素材」按钮 |

## 结果

<Callout type="insight" title="一期数据">
  一期上线后，首次付款率从 **4% 提升至 8%**，近乎翻倍。
</Callout>

我判断提升的关键是**信任感的建立**。视觉参考图是整个流程里用户的第一个 Aha Moment——在这之前，用户只是在和 AI 聊天；参考图出现的那一刻，用户第一次看到了一个具体的交付物预期，信任门槛随之降低。

有一个有趣的数据插曲：直接发长 Prompt 的用户，付费率略高于「聊出来」的用户。推测是因为这类用户在项目上投入的精力更多，付费意愿天然更强。工具包逻辑的好处在于它同样适配这类用户——AI 拿到充分信息后，直接跳过沟通进入生成。

二期的核心目标是提升生成质量，对应指标是用户的编辑次数和编辑时长的前后变化。由于我在数据充分积累前离开了公司，没有拿到具体数值。

## 我学到了什么

### 关于把洞察落地

「服务逻辑 vs 工具逻辑」是一个判断，但停在判断层面什么都落不了。从洞察到落地，中间有大量转换工作——每个节点为什么存在、顺序为什么这样排、沟通原则里每一条的依据是什么。这些不会自然而然，都需要想清楚、说清楚、对齐清楚，才能最终落到产品里。

### 关于跨团队的信息传递

Prompt 交接的摩擦让我意识到：当业务逻辑需要跨团队边界执行时，「我以为你知道这很重要」是不够的。需要把隐性的业务判断显性化——明确哪些必须保留、哪些可以灵活处理，才能让执行侧真正理解意图。这不是信任问题，是认知差异的问题。

### 关于「够用就好」的克制

内容结构的最精简原则，乍听起来像是设计上的妥协，实际上是一个积极的选择。生成更多不等于更好——多余的模块增加生成成本、降低生成速度，用户后期也要花更多精力核对和修改。刚好够用，才是真正对用户有利的设计。
