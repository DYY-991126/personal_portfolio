## 项目背景

这个项目发生在即时设计一次底层重构的阶段。由于产品早期积累了大量历史实现，前端组件层存在明显的维护成本问题，例如组件职责混乱、参数设计失控、复用边界不清晰，部分组件的 props 数量甚至已经膨胀到非常不合理的程度。

随着底层能力重做，团队也同步启动了设计系统建设，希望借这次重构机会，把原本零散、粗糙的界面与组件规则重新收束成一套更清晰、可维护、可持续扩展的系统。我在这个阶段作为主要参与者之一，重点参与了 token、图标、基础组件和复合界面模块的整理与沉淀。

## Design Token

这一层主要展示设计系统里最基础、最稳定、也最容易被复用的部分。对我来说，token 层最重要的价值不是“写了一份规范”，而是把颜色、字体、阴影和间距整理成了团队之后可以持续沿用的基础规则。

### Colors

颜色系统不是简单列一批颜色，而是把基础色、不同层级的色阶，以及浅色 / 深色语境里的对应关系，整理成可复用的 token 结构。

### Typography

字体系统这部分，我更在意跨平台的一致性，以及不同信息层级下的可读性。除了字体本身，还包括字号、字重、行高和内容密度之间的关系。

<Project6TypographyPanel />

### Box Shadow

阴影系统负责处理层级、悬浮和浮层之间的关系。它看起来是细节，但真正落到产品里，会直接影响组件的层次感和信息密度。

<Project6ShadowMotionPanel />

### Spacing

间距系统决定了界面的节奏感，也决定了组件和界面在扩展时是否还能保持稳定。对设计系统来说，这是一层很基础但非常关键的规则。

<Project6LayoutPanel />

## Icon Library

图标库是这次工作里最能体现“细碎但体量很大”的模块之一。它不是一两个图标，而是一整套覆盖不同尺寸、不同入口和不同使用场景的图标资产。

<Project6IconSystemPanel />

## Base Components

基础组件是设计系统里最直接的一层输出。这里我把内容拆成通用组件和业务组件两部分，让阅读路径更清楚，也更容易看出我参与过的工作范围。

<Project6BaseComponentPanel />

## Composite Components

除了 token、图标和基础组件，这套设计系统也沉淀到了更复杂的复合模块中。这里我保留了左侧面板、右侧面板、顶部栏和工具栏这几类更贴近真实编辑器场景的部分，用来证明这套系统最后是怎样落到复杂 UI 里的。
