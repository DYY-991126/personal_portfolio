## 一、项目背景与目标

### 项目发生时机

- 发生时间：产品 2.0 上线前
- 约束：时间紧、任务重、人员配置有限
- Owner：设计侧（无 PM 参与，设计承担部分产品职责）
- 团队组成：设计、前端、后端；能力侧作为相关人做咨询沟通

### 核心目标（两层）

| 层次 | 目标 |
|------|------|
| **第一层** | 让设计适配新的技术架构（Workflow → Agent） |
| **第二层** | 2.0 重做契机下，针对 1.0 因历史原因导致的**界面复杂、交互体验不佳**进行整体简化 |

### 设计资源投入策略

- **视觉**：完全复用 1.0，不做任何修改
- **重点**：(a) AI 对话流；(b) 界面框架简化

## 三、界面框架简化：旧框架 → 新框架

### 3.1 布局改动及核心考量

| 维度 | 旧框架 | 新框架 | 说明 |
|------|--------|--------|------|
| **整体布局** | （待补充旧布局） | 对话面板在左、网站预览在右 | 核心结构变化 |

| 考量点 | 内容 |
|--------|------|
| **感官上的第一印象** | 用户阅读顺序从左到右；对话在左 → 进入页面时直观形成「以对话为驱动的软件」认知 |
| **后续扩展性** | 对话在左时，左侧导航扩展更灵活；可增加功能区（项目、经营数据、Dashboard）；左侧导航 → 对话 → 预览的路径更顺畅 |
| **操作的熟悉性** | 行业内相似产品多为左右布局；降低用户从其他产品迁移过来的学习成本 |

### 3.2 核心交互改变

之前点击网页即可选中元素，并弹出该元素相关的一堆功能项；现在点击网页则是普通的点击效果（如点击链接、滚动等）。用户可以聚焦在更高层次上向 AI 发送指令，并轻易地预览 AI 修改后的效果——这里的「预览」既指「看」，也指「使用」。

## 四、AI 对话流设计逻辑

### 两类 AI 动作

| 类型 | 本质 | 设计重点 |
|------|------|----------|
| **类型 1** | 纯文本输出（思考 / 不思考都一样） | 不需要特殊设计 |
| **类型 2** | 调用 Skill 或 Tool 做事 | 设计重点 |

### 类型 2 的两个设计维度

1. **识别系统能力**：明确系统内有哪些 Skill 和 Tools
2. **确定展示颗粒度**：
   - (a) 真实还原：暴露真实调用
   - (b) 概念包装：技术语言 → 用户可理解的概念（我们选择此方案）

### 具体实施

- **工具归类**：如 ReadFileTool、GraphTool、ListTool 等 → 统一包装成「找文件」
- **任务概念化**：工具 → 任务概念
- **视觉**：任务 → IP 动画 + UI 展示
- **任务类型**：做计划、找文件、生成视频/图片、写代码、调研

### 消息形态（类型 2）

| 状态 | 表现 |
|------|------|
| **普通消息** | AI 文本消息 |
| **进程中** | 「正在执行」卡片（进度数值 + 进度条 + 当前描述） |

### 卡片职责

- **折叠态**：总进度 + 当前最新动态（白话描述）
- **可展开**
- **展开后**：Wegic Studio 风格，各 IP 头像 + 任务状态、工具调用、完成/未完成

### 三层 mapping 链

```
原始 Tool / Skill 调用
    → 工具归类 → 任务概念
    → 任务概念 → 用户可见的 Description
    → 任务概念 → IP + WorkProduct UI
```
