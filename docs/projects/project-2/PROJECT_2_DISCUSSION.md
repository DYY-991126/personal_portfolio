## 项目定位

这是 Wegic 2.0 里程碑目标下 P0 方向 (b) 的执行展开：**生成前沟通** ——在 AI 生成网站之前，先与用户对齐需求，从而让首次生成结果更贴近用户预期。

在 project-1 里已提及这个方向，这里记录完整的设计与执行过程。

## 需求背景补充（重要）

### 生成前沟通服务于更大的目标

生成前沟通不只是为了让首次生成质量更好——它同时也是**生成后 AI 引导**能够有效发挥作用的前提。

生成完成后，AI 会基于用户目标，为用户规划一条「最短发布路径」——即从当前生成结果出发，修改哪些内容、以什么顺序处理，可以最快达到可对外发布的状态。

这条路径要真正有效、被用户采纳，前提是 AI 已经理解了用户的目标。如果没有生成前沟通，用户是 free shot 进来的，AI 对用户的业务目标一无所知，无法给出有针对性的路径建议。

**所以：生成前沟通 → AI 了解用户目标 → 生成后路径引导才有意义。**

### 里程碑目标的两个拆解方向

里程碑：50% 的用户在 1 小时内完成可对外发布的网站。

要达成这个目标，可以从两个层面去做：

| 方向 | 内容 |
|------|------|
| 首次生成质量 | 确保首次生成的网站在大的层面上与用户的预期目标相符合，减少后期修改量 |
| 高效的修改方式 | 支持用户在生成后进行高效修改（如圈选元素评论修改）；负向体验兜底（版本回退） |

生成后 AI 引导是两个方向的连接点：基于生成前沟通获取的上下文，AI 主动规划用户的最短发布路径。

### 两个发起这个需求的动因

1. **里程碑拆解**：围绕里程碑目标，「首次生成质量」是核心杠杆之一，而提升首次生成质量最直接的方式就是生成前对齐用户预期。

2. **产品定位**：Wegic 对标外包服务体验。外包服务的核心是主动性——主动了解客户需求、主动沟通、主动规划。这与工具逻辑（等用户下指令）本质不同。

## 问题背景

旧版本（以及大多数 Web Coding 类产品）的模式：**用户发送生成诉求 → 直接生成网站**。

这种模式在具体业务场景下暴露出两个核心问题：

### 问题 1：成本与用户筛选

- 首次生成收费 $2.99（成本考量 + 筛选高付费意愿用户）
- Few-shot 模式下，用户与产品之间的信任感尚未建立
- 在信任缺失的情况下付费，用户的心理门槛更高

### 问题 2：预期对齐与理解偏差

- 一次性输入缺乏沟通环节
- AI 难以在内容、视觉、功能三个层面与用户真实需求完全对齐
- "输入一堆消息 → 直接执行"容易产生理解偏差：生成内容不完整，或整体思路与用户业务逻辑偏离

## 解题框架

### 核心逻辑

如果一个网站的总任务量恒定，把一部分任务量前置——在 AI 生成之前先与用户沟通好，让 AI 拿到更多信息，从而产出更符合预期的首次结果，降低后期修改量。

**目标**：如果预期满分是 10 分，直接给用户一个 7–8 分的网站，用户只需少量修改即可达到可发布状态。

### "预期"的三个层次

将用户对网站的预期拆解为三层：

| 层次 | 说明 |
|------|------|
| 内容结构层 | 符合网站目标的页面结构与模块组成 |
| 功能层 | 满足业务需求（表单、预约等） |
| 视觉层 | 匹配用户审美预期的视觉风格 |

整个生成前沟通流程，就是围绕这三层逐步对齐而设计的。

### 服务逻辑类比

真实外包团队的标准服务流程——正式做需求前，先与客户进行一轮沟通：了解你想要什么网站、预期是什么、用网站来干什么、行业背景。然后根据复杂度和诉求报价。

**这是标准的服务流程，而非工具逻辑。** Wegic 的定位是外包级建站服务，所以应遵循这个逻辑。

## 沟通流程设计

### 节点设计逻辑

这五个节点并非凭空而来，而是对应了"让 AI 生成网站符合用户诉求"这件事的必要条件：

- 不沟通诉求，AI 没有依据生成内容
- 不对齐内容结构，网站的信息架构会偏离业务目标
- 不确认视觉风格，审美预期无法对齐
- 不收集核心素材（联系方式等事实信息），网站无法真正被用户使用

功能层面（表单、预约等）目前不在生成前流程中对齐，原因见"为什么暂不生成功能"一节。

### 五个节点

```
需求沟通 → 内容结构确认 → 视觉风格确认 → 资料上传 → 生成网站卡片（引导付费）
```

| 节点 | 目标 |
|------|------|
| (a) 需求沟通 | 初步了解用户诉求 |
| (b) 内容结构确认 | 基于沟通生成网站框架，供用户确认 |
| (c) 视觉风格确认 | 生成 9:16 视觉参考图，供用户选择风格 |
| (d) 资料上传（AI to UI 表单） | 用户上传自有核心信息，提升定制感 |
| (e) 生成网站卡片 | 最终生成，引导至付费 |

### 入口：开放式问题

> "你想构建什么？可以跟我说，然后我会一步一步带着你，帮你完成网站。"

### 关键设计调整：从线性流程到工具包逻辑

**初版问题**：直接告诉 AI 按 A→B→C→D→E 的顺序依次走，它对任何诉求都会按部就班——但这不是真实的沟通方式。一个好的沟通者应该因人而异。

**调整方向**：将五个节点重新定义为 AI 的"工具包"，AI 根据上下文自行判断何时调用哪个工具：

- 用户已经发了很长的 Prompt，信息足够充分 → AI 可以直接跳到生成节点
- 用户只发了一句话 → AI 从需求沟通开始逐步引导
- 用户上传了一个文档或链接 → AI 读取后直接提炼，跳过逐轮问答

**另一个推动非线性设计的原因**：用户可能对同一个环节多次表达不满意。比如视觉参考图，观察到有用户说不满意 → AI 重新生成 → 再说不满意 → AI 再次生成。线性流程无法处理这类循环，工具包逻辑天然支持。

**适配不同用户的原则**：

用户进来的方式千差万别——有的一句话，有的一大段，有的上传文件，有的发链接——但 AI 的应对不是按用户类型走死规则，而是基于几条原则灵活判断：

- 用户给的信息越充分，AI 需要主动问的就越少，可以更快进入生成
- 用户给的信息越稀疏，AI 就越需要用渐进式提问引导出来
- 无论用户怎么输入，AI 的目标始终是：在进入生成前，拿到足够的上下文

**有趣的数据洞察**：直接发长 Prompt 的用户，付费率略高于"聊出来"的用户——推测是因为这类用户更重视自己的项目，投入意愿更高，付费倾向天然更强。但这只是插曲，我们并不排斥这类用户，工具包逻辑天然适配他们。

### 一期与二期的拆分

原始规划偏大，执行中发现需要拆分：

- **一期**（12.15 → 1.9 上线）：生成式的沟通流程——让 AI 在生成前与用户逐步对齐诉求
- **二期**（上线后持续迭代）：让沟通内容真正输出成高质量的、符合用户预期的网站

二期的新增内容，一部分来自原始规划，一部分来自线上观察和反馈（如流程不够灵活、Structured Output 稳定性不足）。

### 为什么暂不生成功能

虽然生成网站时直接把功能开通好在逻辑上是更好的，但目前没有做，原因有两点：

1. **计费逻辑复杂**：后端功能依赖 Wegic Cloud，它是独立的按量扣费模块，需用户绑定银行卡，只要表单建立并存于云端就会持续计费。与产品订阅逻辑不同，直接在生成流程中开通，资源消耗、技术方案和持续运行成本都较高。
2. **开发进度**：该需求上线时，后端能力尚在构建中，产品尚未完全支持。

## 沟通策略设计

### 用户画像前提

用户是"业务专家"而非"网站专家"——他们对自己的业务最熟悉，表达门槛最低。因此沟通策略围绕用户的业务展开，而非从专业网站维度切入（不问"你要 A 类还是 B 类网站"这种问题）。

### 五条沟通原则

1. **渐进式提问**：不一次性抛出所有问题
2. **一次只问一个问题**：降低用户认知负担
3. **控制消息长度**：正文消息只包含 1–3 句话
4. **以用户业务为中心**：问的是生意和目标，不是技术选项
5. **边界处理**：办不到就直说，并给替代方案

> 背后的逻辑：不同 AI 的沟通体感差异很大。有的 AI（如 OpenAI）回复长篇大论，信息密度低，用户抓不住重点。简练、核心的表达是整体沟通体验的关键。

### 边界处理

对于我们无法实现的网站类型（如重后端的后台管理系统），Prompt 中明确：若技术上不可行，**如实说明，并给出替代方案**，由用户决定。避免用户付费、生成后才发现做不到。除品牌考量外，还有现实成本：每笔客诉银行判罚 $15，来自市场侧同学的反馈。

### 智能建议功能

由于沟通平均需要 6–7 轮才能进入生成流程，为了降低用户打字成本，当 AI 判断当前问答有可选项时，会在底部展示选项卡，供用户快捷点击。

### 多模态输入支持

| 类型 | 说明 |
|------|------|
| 图片 / 视频 / 文档（doc） | 上传即可被 AI 读取 |
| 链接 / 网站视觉 | 规划中，AI 具备访问能力 |
| 语音 | 按住语音键说话，自动转文字填入输入框 |

**典型场景**：外包客户将需求整理成含网站链接的文档，AI 直接读取，快速获取诉求，避免低效的逐轮问答。

## 内容结构规划

### 为什么不能省略这个节点

视觉风格图（9:16）虽然自带内容结构，但"内容结构规划"节点不可省略，原因有两点：

1. **页面数量不匹配**：内容结构规划出的网站可能是多页的，但视觉风格图通常只展示单页，无法覆盖完整业务需求。
2. **视觉模型的局限性**（Nano Banana 2）：能生成美观的多风格图，但无法生成符合业务要求、能高效达成用户目标的内容结构。

### 内容结构的两个维度

**(a) 网站的说服成本**

网站最终要完成业务目标，必须说服客户。这种说服不只靠文案，更靠结构设计：

| 类型 | 结构特征 |
|------|---------|
| 简洁型（如摄影作品集） | 结构极简，核心是展示作品 |
| 强信任型（如本地商家、SaaS） | 需要产品说明、Feature、信任背书、客户评价等更多模块 |

不同网站类型和建站诉求，对应的内容结构必须不同。

**(b) 业务设计目标：最精简原则**

在高效达成用户业务目标的前提下，采用最精简的网站逻辑——**越简单越好**。

优势分两层：

- **用户层**：网站已能满足目标，用户需要 check 和修改的内容量减少。
- **业务层**：生成成本降低 + 生成速度变快（模块量最少）。

> 注：成本和速度不是核心考量，核心是用户层面——降低"用户用量"。

## 视觉风格确认

### 为什么提供视觉参考图（9:16）

| 原因 | 说明 |
|------|------|
| 建立信任感 | 没有具象参考，用户无法确定交付质量；有了图片，用户心中有预期，信任感得以建立 |
| 技术与成本最优 | 先用生图模型展示网站样子，比直接生成完整网站在技术路径和成本上更优 |
| 达成视觉共识 | "简约""温馨"等词语每个人理解不同；参考图能更有效对齐视觉预期，避免词义偏差 |
| 专业分工 | 用户只提高层次战略诉求，设计层面的决策由专业团队处理 |

### 还原性问题与预期管理

- 现有 AI 技术无法做到百分之百还原参考图，追求极致还原需要工程侧解析，成本高、方案复杂。
- 当前策略：接受非百分之百还原，确保还原 7–8 成，或保证整体风格与参考图一致。
- **用户统一说辞**：参考图是"风格参考图"，而非最终百分之百还原稿。
- 随着生成模型（Gemini 3.1）的迭代，尤其是 Gemini 在视觉理解方面的优势，对参考图视觉特征的把握会越来越精准。

### 技术实现细节

- 为确保视觉风格图中的内容结构与 AI 输出的内容结构一致（避免随机性），有一套专门的工作流。
- 尝试过将 9:16 长图切割为单张图传给 AI 以提升还原性，但综合技术实现成本、token 消耗和未来模型迭代考量，目前传整图。

## 资料上传（AI to UI 表单）

### 核心指导思想

收集的是**事实层面**的信息，而非**宣传层面**的信息：

| 类型 | 举例 | 谁来生成 |
|------|------|---------|
| 事实层面（必须用户提供） | 电话、邮箱、联系方式 | 用户提供，AI 无法生成 |
| 宣传层面（AI 可生成） | Slogan、文案介绍、Logo | AI 生成，用户无需提供 |

### 三条收集原则

1. 收集最少量的资料
2. 这些资料必须是 AI 无法生成的
3. 在网站内是必要的

### 表单交互设计

- 按信息类型分表格展示
- 每项下方有"由 AI 帮你生成"选项——用户没有该项资料时点击标记，AI 自动处理

## 视觉素材处理

> 此模块由其他两位同学负责落地，以下为功能概述，不作为本项目的展示重点。

### 素材优化逻辑

- 用户上传低质量图片（如随手拍），AI 判断是否需要素材效果处理
- **处理原则**：首要确保还原性和真实性，其次才是质量提升

### 视频生成

如果 AI 判断网站需要 Hero 背景视频：
1. 基于用户上传的产品图片，通过"图生图"逻辑生成视频首帧封面
2. 由封面生成整个视频

### 为什么"优化素材"是主动提问项而非默认执行

某些行业（如纺织布料）对图片真实性有极高要求，AI 处理过的图片存在失真风险，可能引发"货不对板"的客诉。因此主动询问，而非自动执行。

## 技术实现：关键决策

### 一期：双 AI 架构 → 二期：模式切换

**一期方案**：新起一个独立的"沟通 AI"，沟通完成后将上下文传递给"生成 AI"。

**问题**：上下文传递必然造成信息损失。在上下文未爆掉的情况下，同一个 AI 处理始终优于跨 AI 交接。

**二期方案**：改为单 AI 的模式切换——
- 需求沟通模式：负责生成前的全部沟通节点
- 生成代码模式：完成模式切换后进入建站生成

好处：上下文完整保留，生成质量更稳定。

### Tool Call 而非 Structured Output

AI 在输出内容结构（草图 UI）和视觉风格图（9:16 参考图）时，不直接输出 Structured Output，而是输出 **Tool Call（工具调用）**。

**决策来源**：上线后发现 Structured Output 不够稳定——AI 生成内容后不显示，或出现各种解析问题。我在一篇分析 Claude 架构的国外文章中看到 Tool Call 方案，在会议上提出，工程侧和 PM 侧都认可（其实大家之前就知道这个方案更好，出于成本原因未执行）。

**后续规划**：所有 AI 输出的非正文类消息，统一改用 AI to UI 方式封装，使用 Tool Call 输出，确保稳定性和速度。

为此专门做了一套 **AI UI 工具**，基于这个逻辑输出，速度和稳定性均有提升。

## 上线后的调整

### 积分成本测算

对话功能需要消耗积分，上线前需要确定给用户分配多少初始额度：

- 测算：整个生成前沟通流程约消耗 60–70 积分
- 策略：在测算值基础上乘以 2，预留足够冗余以覆盖不同 case
- 最终决定：初始额度 **150 积分**

### Prompt 交接卡点

**问题**：将写好的 Prompt 交接给 AI 能力侧时，经常发现内容被改动。

- 能力侧认为细节改动无碍，但从业务层面看这些改动有实质影响
- 根本原因：能力侧缺乏业务敏感度，捕捉不到业务层的细节要求
- 后果：Prompt 调整反复耗费大量时间

**解决方法**：在 Prompt 中明确标注哪些是业务强要求（必须保留），哪些是技术实现层面（能力侧可自行决定），做到清晰分层交接。

## 一期与二期的具体差异

### 一期（1.9 上线）

- 沟通流程固定线性
- 无内容结构确认节点
- 上传模块：AI 发送纯文本提示用户上传，无"AI 帮我生成"选项，无跳过按钮
- 沟通 AI 与生成 AI 分离，上下文通过"总结信息卡片"传递
- 无视觉素材优化和视频生成

### 二期（持续迭代）

**1. 流程灵活化**：线性流程 → 工具包逻辑，AI 根据上下文判断调用

**2. 新增内容结构确认**：

发现内容结构对生成质量影响显著。此前使用 GPT-3.5 时，AI 容易"偷懒"，只生成少量页面内容，当时在 Prompt 里做了硬性规定（必须生成多页，每页 7–8 个模块）。现在去掉硬性限制，改由内容结构确认环节来保证质量——内部评估后，这种方式的生成质量是过关的。

**3. 上传流程重设计**：

| 项目 | 一期 | 二期 |
|------|------|------|
| 形式 | 纯文本提示 | Markdown 表单（左侧素材类型，右侧上传/输入控件） |
| AI 代填 | 无 | 每项有"AI 帮我生成"按钮 |
| 流程出口 | 无跳过 | 新增"暂时跳过"和"提交素材"按钮 |

**4. 单 AI 模式切换**：双 AI 架构 → 沟通模式 / 生成模式的单 AI 切换，上下文完整保留

**5. 视觉素材优化**（另两位同学负责）：自动优化上传图片质量；AI 判断是否需要 Hero 视频，调用视频生成工具，基于产品图"图生视频"

## 数据结果

### 一期数据：首次付款率

首次付款率：**4% → 8%**（近乎翻倍）

**归因判断**：提升的关键是**信任感的建立**，而信任建立的核心是**视觉参考图**——它是用户整个沟通流程中第一个真正可视化的东西，也是第一个 Aha Moment。用户在付费前看到了具体的交付物预期，信任门槛由此降低。

### 二期数据：生成质量

一期衡量的是"用户愿不愿意付费"，二期衡量的是"生成的网站质量有没有提升"，对应指标：

| 指标 | 说明 |
|------|------|
| 一小时内编辑完成率 | 对应里程碑目标：50% 的用户在 1 小时内完成发布 |
| 编辑次数（发布前后对比） | 需求上线前后，用户发布网站所需的编辑次数变化 |
| 编辑时长（发布前后对比） | 需求上线前后，用户从首次生成到发布的总编辑时长变化 |
